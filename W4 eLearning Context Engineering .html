1. What Is Context Engineering?

Standard definition

Context engineering is the practice of providing the right information, constraints, and framing so that an LLM can generate a useful and reliable response.

Context includes:

• Background information
• Constraints (what to include or exclude)
• Assumptions about the task
• The role the model is asked to play

Prompt wording alone is not enough if the context is incomplete or misleading.

2. Common Context Failures

The article highlights that many LLM errors come from context failure, not model weakness.

a. Missing context

The model is asked to answer, but does not know:

• the business goal
• the intended user
• how the output will be used

Result:

• Vague answers
• Generic advice
• Technically correct but useless responses

b. Ambiguous context

The model is given information, but it is unclear or conflicting.

Look for:

• Multiple possible interpretations
• Unclear scope (“optimise”, “improve”, “best”)
• No definition of success

Result:

• Overconfident but wrong answers
• Assumptions that do not match the real problem

c. Irrelevant or excessive context

More information is not always better.

Problems occur when:

• Too much unrelated data is included
• The model cannot tell what matters
• Important signals are buried in noise

Result:

• Distracted responses
• Focus on the wrong factors
• Lower output quality

d. Context mismatch with reality

The context provided does not reflect real-world conditions.

Examples:

• Training-like assumptions applied to deployment
• Clean data assumptions in messy environments
• Ignoring operational constraints

Result:

• Answers that sound good but fail in practice

3. Why Context Failures Matter in ADALL

In this module:

• You use LLMs to frame modelling objectives
• You critique AI-generated answers
• You justify modelling and deployment decisions

Context failures lead to:

• Poor modelling objectives
• Wrong target variables
• Inappropriate metrics
• Weak or misleading project justifications

This is why LLM critique is graded, not just LLM usage.

4. Techniques to Mitigate Context Failures

The article suggests context control, not longer prompts.

a. Clarify the task role

Tell the model who it is and what it is doing.

Example:

• “You are advising a pricing analyst”
• “Focus on problem framing, not algorithms”

b. Define constraints explicitly

State what the model should not do.

Examples:

• “Do not suggest deep learning”
• “Do not assume access to future data”

This reduces hallucinated or irrelevant content.

c. Provide decision context

Explain how the output will be used.

Examples:

• “This will be used to justify a model choice to management”
• “This is for an academic project, not production deployment”

This improves relevance and tone.

d. Ask for structure, not prose

Structured outputs reduce ambiguity.

Examples:

• Bullet points
• Lists
• Tables
• Explicit sections

This makes critique easier and safer.

5. Why Human Judgement Still Matters

The article emphasises:

• LLMs do not know what is “important” unless told
• LLMs cannot judge business impact on their own
• Context design is a human responsibility

LLMs generate text.
You decide whether it makes sense.

This directly links to:

✓ Modelling objective quality
✓ Deployment readiness
✓ Stakeholder communication
✓ Project evaluation criteria

Key Takeaway for You

Context engineering is not about clever prompts.

It is about:

• Giving the right information
• Removing ambiguity
• Limiting assumptions
• Thinking before asking

In ADALL, we want you to takeaway that good data analyst and prompt engineers are not those who get long AI answers, but those who can say:

“This response looks reasonable, but here is what is missing or risky.”

That skill matters in the lab, the practical test, and especially in your project.
